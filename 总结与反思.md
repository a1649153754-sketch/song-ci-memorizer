# 项目总结与反思：宋词背诵辅助工具

## 1. 核心流程还原 (我们是怎么跑通的)

整个项目的开发经历了从“非结构化数据萃取”到“前端交互研发”，再到“数据深度清洗”和“功能迭代”的完整生命周期：

### 第一阶段：从 PDF 到代码 (数据建模)
项目最初的挑战是从《宋词导读作品选》PDF 中提取出结构化的数据。由于 PDF 的排版复杂（包含目录、正文、带星号的重点词目、生僻字注音等），初期我们使用 Python 脚本（配合正则表达式和 `fitz` 库）进行硬提取，初步生成了 `data.js` 作为前端的数据源。

### 第二阶段：视觉呈现与交互 (前端构建)
为了提供沉浸式的背诵体验，我们构建了一个纯前端的静态单页应用 (SPA)。
- 实现了极具现代主义的毛玻璃 (Glassmorphism) 深色薄纱设计。
- 开发了四种核心交互记忆模式：**通读兼纠错模式**、**首字提示**、**随机挖空**、**全隐回想**。
- **踩坑与修复**：在这个过程中，我们遇到了字距过大的渲染 Bug，最终定位到是因为在逐字渲染且使用了 `justify` 对齐方式时，由于没有完整词组导致的排版崩溃，并通过修改 CSS 得到了完美解决。

### 第三阶段：破案与数据修补 (Data Patching)
随着使用的深入，原始数据提取的瑕疵显现了出来：
- **缺失与合并**：发现了漏掉的《满庭芳》，以及因为排版相邻而合并在一起的《天仙子》与《一丛花令》、《暗香》与《疏影》。
- **作者信息丢失**：原脚本由于无法很好地关联目录中的游离作者名，导致作者列全部为空。
- **解决手段**：我们没有选择繁琐的手动编辑，而是编写了 `build_clean_data.py` 自动化补丁脚本，通过反向交叉比对目录 `inspect_output.txt` 和 `normal_works.txt`，用代码一次性自动完成了“拆分、补全、作者映射”的复杂数据重构，最终达成了 88 首星号词目的完美呈现。

### 第四阶段：业务闭环与持久化上线 (功能扩展与 Git)
- 加入了基于浏览器 `localStorage` 的记忆追踪机制，让工具真正具备了个人学习记录的价值。
- 配置并初始化了 Git 环境，编写了标准的 `.gitignore` 和 `README.md`，协助你走通了将代码开源并推送到 GitHub 的标准开发者流程。

## 2. 过程反思：未来可以怎么改进？

虽然我们最终漂亮地跑通了全流程，但回顾过程，有几个环节采用更现代化的方案可以大幅提升效率：

### 改进点一：使用大模型 (LLM) 降维打击 PDF 解析
**痛点**：我们在“清理脏数据”、“拆分合并的词作”、“精准提取字数不等的标题和作者”上花费了很大精力去写复杂的 Python 正则表达式。
**改进方案**：在 2026 年的技术栈下，对于这种非标准排版的 PDF 或文本提取，最佳实践是**不写正则**。应该直接剥离出 PDF 的纯文本内容，然后调用大语言模型的 API（如 Gemini / Claude / OpenAI），给它一个提示词：“请将以下杂乱的宋词文本，严格按照 JSON 格式输出，包含 title, author, content(数组形式) 四个字段”。大模型能凭借语义理解，一次性解决《天仙子》和《一丛花令》的拆分问题，几乎达到 100% 的数据准确率。

### 改进点二：本地持久化架构的选型局限
**痛点**：项目中你曾提出“在网页内直接修改文本并永久保存生效”的需求。由于目前是一个“纯前端 (HTML+JS) + Python 简易静态服务器”的架构，浏览器端的 JS 受到安全沙箱限制，没有任何权限直接改写本地的 `data.js` 物理文件，导致我们最终只能在浏览器内存或 `localStorage` 中模拟持久化。
**改进方案**：如果一开始引入一个极轻量级的后端（例如使用 Node.js 的 Express，或者 Python 的 FastAPI），前端的“修改文本”就可以通过发送一个 HTTP POST 请求告诉后端，让后端脚本直接写入并覆盖 `data.js` 文件，从而真正实现“所见即所得”的本地词库内容永久热更新。

### 改进点三：版本控制的前置
**痛点**：我们在产生了好几个脚本、多次覆写 `data.js` 之后，才在最后一步初始化了 Git。这中间一旦写错了 Python 脚本把数据冲刷掉，是很难快速恢复的。
**改进方案**：最佳的工程实践是：**写下第一行代码前就 `git init`**。在每次我们想要用 Python 去跑批处理修改数据之前，先 `git commit` 一次。这样无论脚本造成了多大的灾难，只需要 `git restore` 就能在一秒内回到安全锚点。

### 总结
这是一个非常经典的“微型全栈痛点项目”。从这个项目中，不仅产出了一个极具实用价值的优美背诵工具，更完整地映射了真实世界软件工程中的“数据清洗 - 呈现层痛点发现 - 自动化脚本弥补 - 功能演进 - 代码托管”的标准缩影。
